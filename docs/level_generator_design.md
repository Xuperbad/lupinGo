# 羊了个羊关卡生成器设计文档

## 1. 基础规则

### 1.1 卡片数据格式
```
[z, y, x, type]
```
- `z`: 层级（0为最底层，数值越大层级越高）
- `y`: Y坐标（行）
- `x`: X坐标（列）
- `type`: 卡片类型（用于匹配消除）

### 1.2 输出格式
```
{[z1, y1, x1, type1], [z2, y2, x2, type2], ...}
```

### 1.3 卡片占位规则
每张卡片占据 **2×2** 的格子空间：
- 卡片位于 `(x, y, z)` 时，实际占据：
  - `(x, y, z)`
  - `(x+1, y, z)`
  - `(x, y+1, z)`
  - `(x+1, y+1, z)`

### 1.4 网格尺寸
- 最大网格：**12×12**（坐标范围 0-11）
- 由于卡片占2×2，卡片左上角坐标范围：**0-10**（确保不超出边界）
- 每层最多可放置：**6×6 = 36** 张卡片（满铺情况）

### 1.5 遮挡规则
- 上层卡片会遮挡其正下方 2×2 范围内的所有下层卡片
- 被遮挡的卡片无法被点击
- 只有当遮挡它的所有上层卡片都被移除后，该卡片才可被点击

## 2. 难度控制参数

### 2.1 核心参数
| 参数 | 说明 | 影响 |
|------|------|------|
| `layers` | 层数 | 层数越多，遮挡关系越复杂 |
| `typeCount` | 卡片类型数量 | 类型越多，匹配难度越大 |
| `cardsPerLayer` | 每层卡片数量 | 卡片越多，选择越复杂 |
| `coverage` | 遮挡率 | 上层对下层的遮挡程度 |

### 2.2 难度等级建议
| 难度 | 层数 | 类型数 | 每层卡片数 | 描述 |
|------|------|--------|------------|------|
| 简单 | 1-2 | 3-4 | 6-12 | 新手入门 |
| 普通 | 2-3 | 5-6 | 12-18 | 适度挑战 |
| 困难 | 3-4 | 7-8 | 18-24 | 需要策略 |
| 地狱 | 4-5 | 9-12 | 24-36 | 极限挑战 |

## 3. 生成器设计思路

### 3.1 核心约束
1. **可解性保证**：每种type的卡片数量必须是3的倍数（三消规则）
2. **公平性**：同一type的卡片不应全部被遮挡在底层
3. **边界检查**：卡片坐标不能超出12×12范围

### 3.2 生成算法流程

```
┌─────────────────────────────────────┐
│ 1. 初始化参数                        │
│    - 设置层数、类型数、难度           │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│ 2. 计算卡片总数                      │
│    - 必须是3的倍数                   │
│    - 每种type分配相等数量             │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│ 3. 逐层生成卡片位置                  │
│    - 从底层(z=0)开始向上生成          │
│    - 检查位置冲突（同层2x2不重叠）     │
│    - 上层可部分遮挡下层               │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│ 4. 分配卡片类型                      │
│    - 确保每种type数量是3的倍数        │
│    - 尽量分散同type卡片到不同层       │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│ 5. 可解性验证（可选）                 │
│    - 模拟游戏过程                    │
│    - 确保关卡可以通关                │
└─────────────────┬───────────────────┘
                  ▼
┌─────────────────────────────────────┐
│ 6. 输出格式化结果                    │
└─────────────────────────────────────┘
```

### 3.3 位置生成策略

#### 策略A：网格对齐（简单）
- 卡片只能放在偶数坐标上 (0,2,4,6,8,10)
- 同层卡片完全不重叠
- 生成简单，但布局较规整

#### 策略B：自由偏移（复杂）
- 卡片可放在任意有效坐标 (0-10)
- 需要检查同层2×2区域是否冲突
- 布局更自然，遮挡关系更复杂

### 3.4 遮挡计算
判断卡片A是否遮挡卡片B（A在上层）：
```
A占据: (ax, ay) ~ (ax+1, ay+1)
B占据: (bx, by) ~ (bx+1, by+1)

遮挡条件: 两个2×2区域有交集
即: |ax - bx| < 2 && |ay - by| < 2 && az > bz
```

## 5. 已确认规则

| 规则 | 确认值 |
|------|--------|
| 消除规则 | **三消**（3张同类型卡片消除） |

## 6. 可解性保证算法（逆向生成法）

### 6.1 核心思路
**不采用正向验证，而是采用逆向生成**，从根本上保证关卡可解。

正向验证问题：
- 需要穷举所有可能的消除顺序
- 计算复杂度极高（NP问题）
- 效率低下

逆向生成优势：
- 模拟"放回"卡片的过程
- 天然保证可解性
- 生成效率高

### 6.2 逆向生成原理
想象游戏结束时，所有卡片已被消除，场地为空。

逆向操作：
1. 每次"放回" **3张同类型** 卡片
2. 放回时，这3张卡片必须都在 **可点击位置**（不被其他卡片遮挡）
3. 后放回的卡片可以遮挡先放回的卡片

玩家正向游戏时：
- 按逆向放置的 **相反顺序** 消除
- 一定存在可行解

### 6.3 逆向生成流程

```
┌─────────────────────────────────────────┐
│ 1. 初始化空场地                          │
│    - 12×12 网格，无任何卡片               │
│    - 准备好所有要放置的卡片类型           │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│ 2. 计算需要放置的轮次                    │
│    - 总卡片数 = typeCount × 3 × n        │
│    - 每轮放置3张同类型                   │
│    - 轮次 = 总卡片数 / 3                 │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│ 3. 循环放置（从底层到顶层）               │
│                                         │
│  for each 轮次:                         │
│    a. 随机选择一个类型                   │
│    b. 寻找3个有效位置                    │
│       - 位置必须当前不被遮挡             │
│       - 或在更高层级                     │
│    c. 放置3张卡片                       │
│    d. 记录放置顺序                      │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│ 4. 验证槽位约束                          │
│    - 确保任意时刻槽位不超7张              │
│    - 若超出，调整放置策略                │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│ 5. 输出结果                             │
└─────────────────────────────────────────┘
```

### 6.4 有效位置判定
卡片位置 `(x, y, z)` 有效的条件：
1. **边界检查**：`0 ≤ x ≤ 10` 且 `0 ≤ y ≤ 10`
2. **同层不冲突**：与同层已有卡片的2×2区域无交集
3. **可点击**：没有更高层的卡片遮挡它

### 6.5 遮挡检测算法
```
函数 isBlocked(card, allCards):
    for each other in allCards:
        if other.z > card.z:  // 只检查更高层
            if |other.x - card.x| < 2 and |other.y - card.y| < 2:
                return true  // 被遮挡
    return false  // 未被遮挡
```

### 6.6 槽位安全验证
为确保玩家不会因槽位满而卡死：
- 模拟正向消除过程
- 跟踪槽位占用情况
- 如果任意时刻槽位 > 7 且无法消除，需要调整

**优化策略**：
- 每轮放置的3张卡片，至少有2张应该在 **顶层或接近顶层**
- 避免同类型卡片全部被深埋

## 7. 实现细节

### 7.1 位置选择策略
为了生成更有趣的关卡：

```
放置第N轮时（共M轮）:
  - 前期轮次（N < M/3）：优先放在中间层和底层
  - 中期轮次（M/3 ≤ N < 2M/3）：随机分布各层
  - 后期轮次（N ≥ 2M/3）：优先放在顶层
```

这样生成的关卡：
- 底层有足够卡片支撑
- 顶层容易开始游戏
- 中间层增加策略深度

### 7.2 类型分配策略
```
每种类型的卡片数 = 3 × k（k为正整数）

分配原则：
  - 每种类型尽量分散到不同层
  - 避免同类型卡片在同一区域堆叠
```
